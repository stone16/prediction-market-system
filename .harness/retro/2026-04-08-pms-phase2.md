# Retro — pms-phase2

## TL;DR

Phase 2 turned the harness from a mock-only self-test into a real
evaluation pipeline that installs, probes, and ranks third-party
prediction-market tools. Six planned checkpoints (P2-01 → P2-06) plus
one bonus follow-on (Phase 3B positions ledger) shipped on
`feature/pms-phase2` over four commits with zero reverts. The test
suite grew from 228 → 323 (+95) while mypy strict stayed clean on
every commit.

The dominant lesson was at the **environment-assumption** layer rather
than the code-correctness layer: the subprocess runner was designed
around `<venv>/bin/python -m pip install <pkg>`, which fails because
`uv venv` produces a venv without pip. The fix was small (route
through `uv pip install --python <venv>` instead) but the assumption
was implicit and would have stayed latent if not for the real
end-to-end smoke run against py-clob-client.

There was no peer review loop on this branch — Phase 2 is mostly
data-entry (YAML candidates + benchmarks) and isolated infrastructure
(subprocess runner, evaluate-all CLI, eval_results aggregator), with
unit tests that monkeypatch the subprocess layer for speed and one
integration test that exercises the real py-clob-client probe end to
end (gated on `PMS_RUN_INTEGRATION=1`).

## Observations

### Error Patterns

1. **[category: tool-env-assumption] `uv venv` does not seed pip.**
   The first version of `subprocess_runner.py` translated
   `pip install <pkg>` into `<venv>/bin/python -m pip install <pkg>`
   on the assumption that the venv would have pip installed by
   default. `uv venv` produces a minimal venv (no pip), so the install
   step crashed with `No module named pip` for every real Python
   candidate. Fix: route both `pip install` and `uv pip install`
   shapes through `uv pip install --python <venv>`, which works
   against any venv layout.

   **Pattern**: when wrapping a third-party tool (`uv venv`,
   `npm init`, `cargo init`), enumerate what the tool actually puts
   on disk before writing code that depends on its output. The
   "obvious" defaults (pip in a venv, lockfile in a npm dir) are not
   universally guaranteed and the failure mode is silent until the
   first real run.

2. **[category: pytest-marker-no-auto-deselect] Markers do not
   self-deselect.** Adding `@pytest.mark.integration` to a test
   doesn't make pytest skip it by default — `-m 'not integration'`
   must be passed explicitly, OR the marker must be combined with
   `pytest.mark.skipif` on an env var. The first version of the
   integration test ran the real py-clob-client install on every
   `pytest` invocation and crashed at the 5-minute timeout. Fix:
   gate the test on `PMS_RUN_INTEGRATION=1` via `pytest.mark.skipif`
   so the marker stays accurate but the default behaviour is "skip".

3. **[category: stale-baseline] Baseline tests broken in committed
   state.** The cloned `main` branch failed to collect
   `tests/test_pipeline.py` because pyproject's `pythonpath = ["python"]`
   did not include the project root, so `from tests._registry_target
   import ...` could not resolve. The baseline ran cleanly on the
   original machine (likely a stale conftest left in the IDE's
   sys.path) but not on a fresh clone. Fix: add `.` to `pythonpath`
   so `tests` is namespace-importable. Lesson: **always run the
   baseline verification command from a fresh clone before assuming
   "228 passed"** — the documented baseline can drift from the
   committed baseline if dev-machine state is load-bearing.

### What Worked Well

1. **Per-checkpoint TaskCreate tracking** kept the work visible
   across the long context. The TodoWrite-style task list also gave
   clear stop/start handoffs between phases.

2. **Parallel tool calls** for independent reads/writes (multiple
   YAML files in one message) cut the wall-clock time roughly in
   half versus serial calls. The 18 candidate YAMLs and the 9
   benchmark YAMLs each shipped in a single batched call.

3. **Whitelist install parser** (`pip install`, `uv pip install`,
   `npm install`, `cargo add` only) was the right call. It rejects
   `make install` / `uv sync` / multi-step shell pipes loudly,
   so a tool with a complex install path surfaces as an
   `UnsupportedCandidateError` in the report instead of silently
   shelling out an opaque command in a sandboxed environment.

4. **Monkeypatched subprocess unit tests** ran 23 tests in <1s while
   covering the full state machine of the runner. The single
   integration test runs the real tool end to end when explicitly
   requested, so the cheap and the expensive checks live side by
   side without forcing every CI run to pay for both.

5. **Real-world validation** during the manual smoke run independently
   confirmed the design: `py-clob-client` actually installed in an
   isolated `uv venv` and passed all three survival gates against
   `https://clob.polymarket.com`. The smoke run also surfaced the two
   failure modes the runner needs to handle gracefully (kalshi's
   Python 3.13 requirement, pmxt's npm-install timeout) — both
   recorded as cleanly eliminated candidates rather than crashes.

6. **Long-only positions ledger** (Phase 3B) was the right scope. The
   spec allowed shorts, but the pms model uses outcome_id-based
   "buy the opposite outcome" semantics, so long-only ledger entries
   plus clamp-on-oversell mirror how the rest of the system thinks
   about exposure. Decimal arithmetic kept the weighted-average math
   exact across the partial-fill test cases.

## Recommendations

### Proposal 1: Verify isolated-env tooling assumptions before wrapping

- **Pattern**: tool-env-assumption
- **Severity**: medium
- **Status**: Proposed
- **Root cause**: The subprocess runner's first version was written
  on the assumption that `uv venv` produces a pip-equipped venv.
  It does not. The bug was discovered during the first real py-clob-client
  install and fixed in one line, but it would have been caught earlier
  by a 30-second sanity check: `cd /tmp/test && uv venv .v && .v/bin/python -m pip --version`.
- **Drafted rule text**:
  ```
  ## Verify Tool Output Before Wrapping (🟡 IMPORTANT)

  When writing a wrapper around a third-party CLI tool that produces
  files or directories on disk (`uv venv`, `npm init`, `cargo init`,
  `python -m venv`, etc.), enumerate what the tool actually creates
  before writing wrapper code that depends on the output:

  1. Run the tool against a temp dir manually and `ls -la` the result.
  2. Note which "obvious" files are missing (pip in a uv venv, package.json
     entries in an npm init, etc.).
  3. Write the wrapper against the actual layout, not the assumed
     layout.

  The failure mode is silent until the first real invocation; unit
  tests with mocked subprocess calls will pass either way.
  ```
- **Issue-ready**: true

### Proposal 2: Fresh-clone baseline verification

- **Pattern**: stale-baseline
- **Severity**: medium
- **Status**: Proposed
- **Root cause**: The pms-v1 baseline was documented as "228 passing"
  but failed to collect on a fresh clone because pyproject's
  `pythonpath` was missing `.`. The original machine had IDE state
  that masked the bug. The continuation guide (Step 3) tells the next
  agent to run the verification commands but does not call out the
  fresh-clone requirement.
- **Drafted rule text**:
  ```
  ## Fresh-Clone Baseline Verification (🟡 IMPORTANT)

  When picking up a project on a new machine, ALWAYS run the
  documented baseline verification commands from a fresh clone in
  a fresh shell before assuming the documented baseline holds.
  Dev-machine state (IDE plugins, stale venv, sys.path injections)
  can hide config bugs that bite the next contributor.

  If the baseline fails on a fresh clone, fix the config (not the
  test) and commit the fix as the first commit on the feature branch
  with a `fix(tests):` or `fix(build):` prefix. Do not start feature
  work against a broken baseline.
  ```
- **Issue-ready**: true

### Proposal 3: Pytest markers must be self-deselecting OR env-gated

- **Pattern**: pytest-marker-no-auto-deselect
- **Severity**: low
- **Status**: Proposed
- **Root cause**: Adding `@pytest.mark.integration` to a test does
  not make pytest skip it by default. Either the project must add
  `addopts = ["-m", "not integration"]` to pyproject (which then
  affects every test invocation), or each integration test must
  combine the marker with `pytest.mark.skipif` on an env var. The
  env-var pattern is more granular and was the chosen fix here, but
  it's a footgun that future contributors will trip over.
- **Drafted rule text**:
  ```
  ## Integration Test Default-Skip Pattern (🟢 RECOMMENDED)

  When adding a `@pytest.mark.integration` (or `@pytest.mark.slow`)
  test that requires network / real subprocess / external state,
  combine the marker with an env-var skipif so the default `pytest`
  invocation does not run it:

      @pytest.mark.integration
      @pytest.mark.skipif(
          os.environ.get("PMS_RUN_INTEGRATION") != "1",
          reason="set PMS_RUN_INTEGRATION=1 to run integration tests",
      )

  The marker keeps `pytest -m integration` working as expected; the
  skipif keeps the default invocation fast and offline.
  ```
- **Issue-ready**: true
