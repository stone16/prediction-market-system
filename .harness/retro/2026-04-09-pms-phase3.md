# Retro — pms-phase3

## TL;DR

Phase 3 closed out the four "alternative phases" the continuation
guide flagged after Phase 2: 3A live read paths, 3C LLM correlation
refinement, 3D retro rule adoption, and 3E Rust scaffolding. All
four shipped first-try with zero iterations on five commits, taking
the test suite from 323 → 352 (+29) and the mypy surface from 61 →
65 source files. No reverts; mypy strict stayed clean on every commit.

The dominant lesson was scope discipline: each of the four phases
sat at a different ambition tier, and trying to push all four to
"production complete" would have re-introduced the risk patterns the
pms-v1 retro flagged. Instead each phase shipped at the right
increment for its risk profile:

- **3A** shipped *real* implementations for the two deferred
  ConnectorProtocol methods (`stream_prices`, `get_historical_prices`)
  but explicitly **did not** add a `submit_order` write path. Read
  paths against public Polymarket endpoints have zero blast radius;
  write paths need real wallet credentials and an explicit user opt-in
  that's outside the scope of this branch.
- **3C** shipped a fully-tested LLM refinement layer with
  ``anthropic`` as an **optional** dep — install with
  ``pms[llm]`` — and a stubbable provider interface so tests never
  burn tokens or hit rate limits.
- **3D** promoted retro proposals to a project-level ``CLAUDE.md``
  but explicitly **did not** edit the user's global ``~/.claude/CLAUDE.md``
  (out of repo scope; that's a manual user step).
- **3E** scaffolded the three Rust crates with PyO3 stubs + Python
  fallback layer + 7 contract tests, but explicitly **did not** wire
  the Rust build into ``uv sync``. The Rust toolchain is not
  installed on this machine and the canonical Python implementations
  cover every path; the scaffolding clears the way for a future task
  to actually build the crates without reshaping anything.

## Observations

### What Worked Well

1. **Per-checkpoint commits.** Phase 3 produced 5 commits across 4
   checkpoints (3D, 3A, 3C, 3E + 1 lockfile chore). Each commit is
   independently reviewable, mypy-clean, and test-green. Compared to
   the single ``feat(phase2)`` commit that bundled P2-01 → P2-05 in
   one diff, the per-checkpoint shape makes it possible to bisect a
   regression to a specific feature in seconds.

2. **Scope-bounding by risk profile.** Each phase landed at the right
   increment for its blast radius:
   - 3A: read paths (low risk) → fully implemented
   - 3C: LLM with optional dep (zero blast radius) → fully implemented
   - 3D: project rules (zero risk) → fully implemented
   - 3E: Rust scaffolding (no toolchain available) → scaffolded only,
     with the Python fallback contract pinned by tests

   Pushing 3E to "fully built" would have required installing the
   Rust toolchain, building three PyO3 crates, and wiring them into
   ``uv sync`` — all of which would have been net-negative without
   actual hot-path implementations to accelerate.

3. **Stub-classifier pattern for Phase 3C.** The
   ``LLMClassifierProtocol`` lets unit tests inject a deterministic
   ``_FakeClassifier`` that returns canned responses. The 16 tests
   cover the refiner happy path, pass-through behaviour, exception
   fallback, JSON parsing edge cases, and the Anthropic provider's
   text-extraction + classify_pair plumbing — all without ever
   importing the real ``anthropic`` package. Same pattern as the
   subprocess runner from Phase 2: the *contract* gets exhaustive
   coverage cheaply, the *provider* is exercised separately by the
   integration test (gated on credentials).

4. **Phase 3D rule promotion validated retroactively.** Three of the
   eight rules promoted to ``CLAUDE.md`` in 3D had **already** been
   followed implicitly across Phase 2 + Phase 3:
   - "comments are not fixes" — every fix in this branch changed
     runtime behaviour, never just added a NOTE.
   - "lifecycle cleanup on all exit paths" — the
     ``SubprocessRunnerFactory.cleanup_all()`` call lives in a
     ``finally:`` block in both the single-module and ``evaluate-all``
     CLI paths, and the LLM refiner falls back gracefully on
     classifier exceptions.
   - "runtime behaviour > design intent" — the deferred-method tests
     in Phase 3A were *replaced* with real-implementation tests
     rather than left as "documents the limitation" cases.

   That's the validation signal for promotion: rules that the
   author has been following without conscious thought are good
   candidates for ``active`` status because they've already passed
   the "actually useful" filter.

5. **PyO3 0.22 + Cargo workspace shape.** The three crates share
   ``[workspace.package]`` metadata for edition / MSRV / license /
   repo, so adding a fourth crate later only needs a `Cargo.toml`
   declaring the package name + lib metadata + the inherited
   workspace fields. No duplication, no version skew.

### Error Patterns

(None genuinely new this branch — Phase 3 stayed inside the rules the
two prior retros codified. The closest call was the lockfile drift
in 3E which is documented separately below as a low-severity
chore-pattern observation.)

1. **[category: lockfile-drift-on-optional-dep] Optional-dep adds
   touch the lockfile even without ``uv sync``.** Phase 3C added
   ``anthropic`` as an optional dep in pyproject.toml; the next
   ``uv run pytest`` invocation regenerated ``uv.lock`` with the
   resolved transitive dependency tree because uv resolves the lock
   for **all** declared extras by default. The 3C commit shipped
   without the lockfile change because the regeneration happened on
   the next invocation, after the commit landed. The fix was a
   trailing ``chore(deps)`` commit; cleaner would have been to run
   ``uv sync`` (or ``uv lock``) after the pyproject edit and stage
   the lockfile change as part of the same commit.

   **Pattern**: when adding optional deps in pyproject, the next
   `uv` invocation will refresh the lockfile. Stage the lockfile
   change in the same commit as the pyproject edit.

## Recommendations

### Proposal 1: Stage lockfile changes with pyproject edits

- **Pattern**: lockfile-drift-on-optional-dep
- **Severity**: low
- **Status**: Proposed
- **Root cause**: Phase 3C committed pyproject.toml's optional-dep
  declaration without the corresponding lockfile regeneration. The
  next test invocation produced the lockfile change as a side
  effect, requiring a trailing chore commit.
- **Drafted rule text**:
  ```
  ## Stage Lockfile Changes With pyproject Edits (🟢 RECOMMENDED)

  When adding or modifying entries in pyproject.toml that affect the
  dependency resolver — adding/removing/upgrading a dep, declaring
  an optional-dep group, changing python-version constraints — run
  ``uv lock`` (or ``uv sync``) immediately after the edit and stage
  the resulting lockfile change in the same commit.

  Reason: uv refreshes the lockfile lazily on the next invocation,
  so the diff produced by your change will surface as a trailing
  "chore(deps)" commit on the next run otherwise. This is harmless
  but it makes the commit history less self-contained.
  ```
- **Issue-ready**: false (low severity, single occurrence — monitor
  for recurrence before adopting)
