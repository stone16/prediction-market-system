# Retro — pms-v1

## TL;DR

Checkpoint pipeline executed cleanly (9/10 first-try PASS, 1 genuine iter-2
bug fix on CP08, zero reverts). The review-loop then exposed a second layer
of defects the checkpoint-scoped evaluator could not see: **cross-module
wiring, lifecycle cleanup, and domain-math correctness under interaction
of features implemented across multiple checkpoints**. 15 findings across
5 iterative rounds + 1 final consensus round, 14 accepted and fixed. The
loop worked — every round was load-bearing, each surfacing bugs introduced
or missed by the previous round's fix.

The dominant lesson is not about code quality per se. It is about
**rejection hygiene**: in round 1, four rejections were written with
handwavy "documented limitation" / "out of scope" reasoning, and three of
them turned out to be real bugs the peer correctly insisted on. The
rejection bar must rise to match the acceptance bar.

## Observations

### Error Patterns

1. **[category: review-rejection-hygiene] Weak initial rejections
   masquerading as scope discipline.** In round 1, I rejected f2, f6, f9
   citing "documented optional path" / "has a NOTE comment" / "dedup
   handled at executor". Codex re-insisted on all three in round 2 with
   concrete repro logic:
   - f2: "the `CorrelationDetector.on_correlation_found()` path is not
     wired anywhere in the repo" — a production dead-code claim, not an
     opinion.
   - f6: "A NOTE comment does not change the externally visible bug" — a
     behavioural claim.
   - f9: "`OrderExecutor` only deduplicates retries of the same
     `order_id`, while `ArbitrageStrategy` emits fresh IDs" — a precise
     logic trace.

   All three rejections were accepted as fixes in round 2. The pattern:
   **my rejections argued from design intent; the peer argued from
   runtime behaviour**. Runtime behaviour won every time.

2. **[category: domain-math-piecewise] Non-linear exposure functions
   fail under mixed inventory states.** f1 / f10 / f12 / f15 are all the
   same underlying issue at different depths: the exposure delta of a
   sell order is not a single linear function of size; it has a break
   point at `size == available_long_inventory`. Up to that breakpoint the
   sell reduces long exposure; beyond it, every additional unit adds
   short exposure. Round 1 treated it as linear (all sells reduce). Round
   2 made it inventory-aware but still linear inside each branch. Round
   3 finally made `_max_fillable_sell_size` piecewise. Final consensus
   (f15) caught that the `strictly_reducing` shortcut bypassed the cap
   check when a long→over-cap-short flip straddled the break point.

   **Pattern**: when implementing a function over a domain with piecewise
   semantics (inventory regimes, fill tiers, leverage bands), the initial
   implementation almost always assumes the easier regime and ships
   tests that only exercise that regime.

3. **[category: lifecycle-cleanup-exit-paths] State mutations scoped to
   a single exit path.** f13 / f14 are both "opportunity dedup state
   leaks because `clear_opportunity()` is only called on the happy
   path". Round 2 fix wired `clear_opportunity` on normal completion.
   Round 3 (f13) pointed out it was never called on reject/skip paths.
   Round 4 (f14) pointed out it was still skipped on executor exceptions
   — needed a `try/finally`. The fix that makes cleanup robust is the
   same as the fix that makes resource handling robust: **cleanup must
   be scoped to the widest control structure that owns the state
   (`try/finally`), not to any individual exit branch**.

4. **[category: cross-checkpoint-integration] Checkpoint-scoped
   evaluation cannot detect dead wiring between earlier and later
   checkpoints.** CP05 implemented `CorrelationDetector`. CP08/CP09
   implemented the pipeline. Neither evaluator could notice that the
   detector was never invoked by the pipeline, because each checkpoint
   passed its own AC in isolation. The evaluator is structurally blind
   to "feature X exists but is dead code downstream" because its scope
   is bounded by the checkpoint's file set.

5. **[category: project-ide-tooling-drift] Pyright IDE false positives
   on every checkpoint.** Every checkpoint surfaced Pyright warnings
   about "Import 'pms.X' could not be resolved" despite
   `uv run mypy --strict` passing cleanly. Pyright was not using the
   uv-managed venv. The canonical check was always mypy strict and all
   Pyright warnings were safely ignored — but this added noise to
   every evidence bundle. **Non-blocking, but a recurring distraction.**

6. **[category: magnitude-overrun-tests] M-effort checkpoints
   consistently exceed the 3× insertion guardrail, driven by test LOC.**
   CP01 (3.1×), CP03 (~2.5×), CP06 (1399 vs 450), CP07, CP08 (2.59×) all
   overran. Every overrun was tests. Production code in each case landed
   at or just inside the M ceiling. The magnitude gate is conflating
   "feature size" with "test size". Every overrun was `auto_resolvable:
   true` because the signal was informational.

### Rule Conflict Observations

1. **Co-authored-by trailer conflict (CP01).** The harness system prompt
   instructed the Generator to add `Co-Authored-By` lines to commits; the
   user's global git rules and the Generator agent prompt both forbade
   them. The Generator resolved by following the agent prompt (which
   explicitly states it overrides system-level instructions) and
   documented the decision in Rule Conflict Notes. **This worked
   correctly** — the conflict was surfaced, resolved with a stated
   precedence, and the chosen resolution matched user intent. It is
   worth codifying the precedence rule once instead of re-deriving it at
   every checkpoint.

2. **Spec ambiguity on `bool` vs `int` in YAML config (CP02).** The spec
   said `timeout_seconds: int`. Python's `isinstance(True, int)` is
   `True`, so naively loading `timeout_seconds: true` would silently
   parse as `1`. Generator added a `bool` pre-check before the `int`
   check and documented the judgment call. Non-conflict, but an
   interpretive gap the spec did not close.

### What Worked Well

1. **Checkpoint scope discipline was excellent.** 22 commits, zero
   reverts, zero cross-checkpoint file modifications (all CP08+
   `git diff --diff-filter=M` against earlier SHAs were empty). The
   "atomic growth" invariant held across the entire task.

2. **First-try PASS rate was 9/10 (90%).** The only iteration was CP08
   iter-2, and it was a genuine bug (partial-fit total-exposure
   check) — not an over-strict evaluator or a retry over flakiness.

3. **Goal-bound iteration during the review loop.** Every round-N fix
   touched only the files named in the round-N findings. No opportunistic
   refactoring. No "while I'm here" changes. Commit messages named the
   specific finding (`f10`, `f11`, …) so the mapping between review
   output and code change was fully traceable.

4. **Peer review loop produced genuinely new information across all 5
   rounds + final consensus.** Round 1 findings were architectural
   (missing wiring, wrong semantics, unenforced timeouts). Rounds 2–5
   found bugs **inside the previous round's fixes**. This is the loop
   working as designed — convergence, not ceremony.

5. **Final consensus round (fresh Codex session) caught f15**, an
   edge case that had slipped past 5 rounds of iterative review. Worth
   keeping the "fresh session for final consensus" protocol step: it
   resets context bias.

6. **Test counts climbed monotonically**: baseline 205 → final 228
   (+23) across the review loop. No test deletions, no test weakening to
   make fixes pass.

The "should have caught" bucket clusters around two weaknesses:
**domain-math symmetry** (f1, f10) and **documenting instead of fixing**
(f6, f11, f14 partially). Both are process issues the generator can
correct with explicit checks before marking a task done.

The "only apparent from scenarios" bucket is genuinely the peer's value
— it's what iterative cross-model review is for, and trying to eliminate
it in the generator phase would be the wrong optimisation target.

## Recommendations

### Proposal 1: Review-loop rejection hygiene

- **Pattern**: review-rejection-hygiene
- **Severity**: high
- **Status**: Proposed
- **Root cause**: In round 1, 4/9 findings were rejected with reasoning
  that amounted to "this is documented" / "this is out of scope" /
  "this is handled elsewhere". Three of those four rejections were
  wrong and the peer correctly re-insisted. The failure mode is
  writing a rejection as an opinion about design intent rather than a
  falsifiable claim about runtime behaviour.
- **Drafted rule text**:
  ```
  ## Review-Loop Rejection Discipline (🔴 CRITICAL)

  When rejecting a peer reviewer's finding during a review loop, the
  rejection MUST include:

  1. A concrete counter-trace: name the file, line range, and specific
     runtime behaviour that contradicts the finding. "Documented in
     NOTE" and "handled elsewhere" are NOT valid rejections.
  2. A named call-site: if claiming a behaviour is "handled elsewhere",
     link to the exact file:line where the handling occurs and show
     that the caller actually hits it.
  3. A reproducible witness: if claiming an edge case cannot occur,
     either cite a test that proves it or write one in the rejection.

  Rejections failing this bar must be re-classified as "accept with
  minimal fix" before the round closes. "This is out of scope" is
  only a valid rejection when the scope boundary is explicit in the
  task spec — not when it is implicit in the current checkpoint.
  ```
- **Issue-ready**: true

### Proposal 2: Piecewise-domain verification for non-linear functions

- **Pattern**: domain-math-piecewise
- **Severity**: high
- **Status**: Proposed
- **Root cause**: Four separate findings (f1, f10, f12, f15) were the
  same bug at different depths: treating the sell-exposure function as
  linear when it has a break point at available inventory. Each fix
  assumed the simpler regime and shipped tests only for that regime.
- **Drafted rule text**:
  ```
  ## Piecewise-Domain Functions (🟡 IMPORTANT)

  When implementing a function whose output has piecewise semantics
  over its input domain (inventory regimes, fill tiers, leverage
  bands, signed vs unsigned inputs, covered vs uncovered trades):

  1. Identify and name every break point in the domain before writing
     code. Document them in a docstring `# Break points: ...`.
  2. Write at least one test input per regime AND at least one test
     input that straddles each break point (e.g. `size = inventory`,
     `size = inventory + 1`).
  3. Derive each regime from first principles, not by sign-flipping or
     adapting another regime's formula. Copy-adapting piecewise code
     is a known source of symmetry bugs.
  4. When fixing a bug in a piecewise function, re-derive all regimes.
     A fix that only corrects one regime is a ticking bomb for the
     other.
  ```
- **Issue-ready**: true

### Proposal 3: Lifecycle cleanup on all exit paths

- **Pattern**: lifecycle-cleanup-exit-paths
- **Severity**: high
- **Status**: Proposed
- **Root cause**: `clear_opportunity` was added to the happy path
  (round 2), then rejected/skipped paths (round 3), then the exception
  path (round 4). Three rounds for one bug. The underlying issue is
  that lifecycle cleanup was scoped to individual exit branches
  instead of the widest control structure that owns the state.
- **Drafted rule text**:
  ```
  ## Stateful Dedup & Lifecycle Cleanup (🟡 IMPORTANT)

  When introducing stateful dedup, tokens, locks, or any resource
  acquired mid-function that must be released:

  1. Acquire and release must be wired in the SAME commit. A PR that
     adds `acquire()` without the corresponding `release()` is
     incomplete, regardless of test status.
  2. Release MUST be in a `try/finally` (or equivalent context
     manager) scoped to the widest block that owns the state — not
     distributed across individual exit branches.
  3. Every early-return path must be checked against the cleanup
     contract: reject, skip, exception, success — all four paths
     must release.
  4. Evaluator Tier 2 review MUST grep for every `acquire` / `add` /
     `register` call in the diff and verify each has a matching
     cleanup wired in a control structure that covers all exit
     paths.
  ```
- **Issue-ready**: true

### Proposal 4: "Documenting is not fixing"

- **Pattern**: document-instead-of-fix
- **Severity**: medium
- **Status**: Proposed
- **Root cause**: f6 (pagination cursor ignored, added a NOTE comment)
  and f11 (P&L semantics wrong, added a docstring) and the iter-1 fix
  for f1 (added documentation explaining the limitation instead of
  fixing it). Adding a comment is not a fix for a behavioural bug.
- **Drafted rule text**:
  ```
  ## Comments Are Not Fixes (🟡 IMPORTANT)

  When a review finding describes incorrect runtime behaviour
  (wrong values, missing calls, silent truncation), the fix MUST
  change runtime behaviour. Adding a docstring, a NOTE comment, a
  TODO, or an "informational" caveat is NOT a valid fix for a
  behavioural finding.

  A behavioural finding is closed only when:
  - The code produces different output for the original failing
    input, AND
  - A new test locks in the corrected behaviour.

  Documentation-only resolutions are reserved for findings that
  are themselves about documentation (missing docstrings, unclear
  parameter contracts, stale comments).
  ```
- **Issue-ready**: true

### Proposal 5: Cross-checkpoint integration trace after final checkpoint

- **Pattern**: cross-checkpoint-integration
- **Severity**: medium
- **Status**: Proposed
- **Root cause**: f2 (`CorrelationDetector` unwired) was invisible to
  per-checkpoint evaluators because each checkpoint passed its own AC
  in isolation. Dead-wiring across checkpoint boundaries is a
  structural blind spot of the harness.
- **Drafted rule text**:
  ```
  ## Cross-Checkpoint Integration Verification (🟡 IMPORTANT)

  After the final checkpoint of a multi-checkpoint task, and before
  review-loop kicks in, the Evaluator MUST run an integration trace:

  1. List every class and public function introduced in checkpoints
     CP01..CP(N-1) that is supposed to run in production (exclude
     explicitly test-only helpers).
  2. For each, grep the production code path from CP(N) (typically
     the pipeline / orchestrator / entrypoint) and verify at least
     one call-site hits it.
  3. Any production class with zero call-sites in the final pipeline
     is flagged as `dead-wire` and must be addressed before the
     review loop begins — not left for the peer to discover.

  This check runs ONCE at the end of the task, not per checkpoint.
  ```
- **Issue-ready**: true

### Proposal 6: Magnitude guardrail — production vs test LOC split

- **Pattern**: magnitude-overrun-tests
- **Severity**: low
- **Status**: Monitoring
- **Root cause**: Five M-effort checkpoints overran the 3× insertion
  guardrail purely on test LOC. Every overrun was `auto_resolvable:
  true`. The guardrail is technically firing correctly but is producing
  noise on every M-tier checkpoint, which weakens the signal for real
  overruns.
- **Proposed action**: Not a CLAUDE.md rule — this is a harness tuning
  suggestion. Evaluate splitting the magnitude budget into
  `production_insertions` (hard gate) and `test_insertions` (soft
  guideline, warn only). Alternatively, raise the M test-LOC ceiling
  to 2× production LOC and keep the production LOC gate unchanged.
- **Issue-ready**: false (monitoring, no rule to draft yet)

### Upgrade to Principle

### Principle: Runtime Behaviour > Design Intent

- **Pattern**: architectural
- **Severity**: high
- **Status**: Proposed
- **Root cause**: The review-loop rejection pattern reveals a deeper
  bias: when defending code, the generator naturally argues from
  "what I meant the code to do"; when attacking code, the peer argues
  from "what the code actually does when run". In every disagreement
  across 5 rounds, runtime behaviour won.
- **Drafted principle text**:
  ```
  ## Runtime Behaviour > Design Intent (Core Principle)

  When evaluating whether code is correct — whether reviewing your
  own, defending against a peer, or accepting a finding — ALWAYS
  argue from runtime behaviour, never from design intent.

  Runtime-behaviour claims are: "when this function is called with
  X, it returns Y; the spec requires Z; therefore bug". They are
  falsifiable, reproducible, and terminate the discussion.

  Design-intent claims are: "this is meant to be handled at the
  X layer", "the Y pattern documents this limitation", "the caller
  is expected to Z". They are unfalsifiable, they depend on context
  the peer may not share, and they are almost always wrong under
  scrutiny.

  If you catch yourself writing "the intent is...", rewrite the
  sentence to start with "when called with...".
  ```
- **Issue-ready**: true

### Rule Conflict Resolution

### Proposal 7: Codify commit-message precedence

- **Pattern**: rule-conflict-precedence
- **Severity**: low
- **Status**: Proposed
- **Root cause**: Every CP01 checkpoint has to re-derive the
  `Co-Authored-By` precedence: harness system prompt adds it, user's
  global git rules forbid it, Generator agent prompt forbids it. The
  Generator correctly resolved this three times (CP01 output-summary,
  Rule Conflict Notes, actual commit), but the derivation is wasteful.
- **Drafted rule text**:
  ```
  ## Commit Message Precedence (🟡 IMPORTANT)

  Commit message attribution follows this precedence (highest wins):
  1. User's global git rules (~/.claude/CLAUDE.md "Git Rules")
  2. Generator agent system prompt instructions
  3. Harness system prompt defaults
  4. Anthropic/upstream template defaults

  Specifically: the user's global rule "Never add `Co-Authored-By`
  lines" overrides any harness or template that adds them. Do not
  re-derive this at every checkpoint; it is settled.
  ```
- **Issue-ready**: true

### Skill Defect Flags

### Skill Defect 1: Pyright IDE configuration drift

- **Skill**: harness / dev environment setup
- **Severity**: low (noise, not blocking)
- **Observation**: Every checkpoint surfaced Pyright warnings about
  "Import 'pms.X' could not be resolved" despite `uv run mypy --strict`
  passing cleanly. Pyright was not picking up the uv-managed venv. The
  canonical check was always mypy and all Pyright warnings were
  ignored, but evidence bundles carry the noise.
- **Recommended fix**: either (a) add a `pyrightconfig.json` to the
  task bootstrap template that points at `.venv`, or (b) document in
  the Generator skill prompt that Pyright is non-canonical and its
  warnings should be explicitly suppressed in evidence if pysy strict
  is clean.
- **Action**: flag for human review. **Not a CLAUDE.md rule** — this is
  a harness template defect, not a behavioural rule for Claude.

### Skill Defect 2: Magnitude gate firing on test LOC

- **Skill**: harness evaluator / magnitude check
- **Severity**: low (auto-resolvable every time)
- **Observation**: 5 of 10 M-effort checkpoints overran the 3×
  magnitude guardrail on insertions alone, every one driven entirely
  by test code. The gate is correctly identifying "more code than
  expected" but not distinguishing between production and test code,
  which makes the signal almost meaningless for M-tier checkpoints
  with thorough test coverage.
- **Recommended fix**: split the magnitude budget into production and
  test components, or raise the soft ceiling for test insertions to
  2× production LOC.
- **Action**: flag for human review, see Proposal 6.
